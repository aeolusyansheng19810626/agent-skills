# 🤖 ルーターエージェント - 統合チャットインターフェース

LLMによるユーザー意図の自動認識と専門スキルモジュールへのルーティングを行うインテリジェントAIエージェントシステム。統一されたチャットインターフェースでリアルタイムにスキル呼び出しステータスを表示します。

## 🌟 コア機能

- **🎯 インテリジェントルーティング**: ユーザー入力に基づいて適切なスキルを自動選択
- **🔧 リアルタイム表示**: 実行時に「🔧 呼び出し中: xxx_skill」を表示
- **📦 モジュラースキル**: skill.mdとskill.pyを作成するだけで新しいスキルを簡単に追加
- **💬 Streamlit UI**: 美しくレスポンシブなチャットインターフェース
- **🔄 ストリーミングレスポンス**: スキル実行結果をリアルタイムでストリーミング出力
- **🌐 多言語サポート**: 中国語、日本語（半角カタカナ）、英語の3言語インターフェース切り替え
- **🎨 パープルグラデーションデザイン**: モダンなパープルグラデーションテーマと最適化されたUIレイアウト

## 🚀 クイックスタート

### 1. 環境要件

- Python 3.8以上
- Groq APIキー ([取得先](https://console.groq.com/keys))
- Tavily APIキー ([取得先](https://tavily.com/))

### 2. インストール手順

```bash
# リポジトリをクローン
git clone <your-repo-url>
cd agent-skills

# 仮想環境を作成
python -m venv venv

# 仮想環境をアクティベート
# Windows:
venv\Scripts\activate
# macOS/Linux:
source venv/bin/activate

# 依存関係をインストール
pip install -r requirements.txt
```

### 3. 環境変数の設定

```bash
# 環境変数テンプレートをコピー
cp .env.example .env

# .envファイルを編集してAPIキーを追加
# GROQ_API_KEY=your_groq_api_key_here
# TAVILY_API_KEY=your_tavily_api_key_here
```

### 4. アプリケーションの実行

```bash
streamlit run app.py
```

アプリケーションはブラウザで `http://localhost:8501` に開きます

## 🔧 動作原理

### ルーターエージェントフロー

```
ユーザー入力 → ルーターエージェント → Groq LLM → 意図認識 → 単一スキル/パイプラインを返す → パイプラインエグゼキューター → ストリーミング出力結果 → UI表示
                                                                     ↓
                                                             動的評価（オプションでスキル追加）
```

### コアコンポーネント

1. **スキルローダー** (`skill_loader.py`)
   - `skills/`ディレクトリを自動スキャン
   - `skill.md`ファイルを解析してメタデータを取得
   - スキルレジストリを構築
   - 動的スキルインポートを提供

2. **ルーターエージェント** (`router.py`)
   - すべてのスキル説明をLLMプロンプトに注入
   - Groq LLMを使用して意図認識（5層モデルフォールバックをサポート）
   - 3つの形式を返す：
     - 単一スキル：`{"skill": "name", "params": {...}}`
     - シリアルパイプライン：`{"plan": [{"skill": "name1"}, {"skill": "name2"}]}`
     - パラレルパイプライン：`{"plan": [{"parallel": [{"skill": "name1"}, {"skill": "name2"}]}]}`
   - 完全な簡体字中国語推論プロセス
   - マッチしないクエリのフォールバックロジックを処理

3. **パイプラインエグゼキューター** (`pipeline.py`)
   - シリアル実行をサポート：順番にスキルを実行し、自動的にコンテキストを渡す
   - パラレル実行をサポート：ThreadPoolExecutorを使用して並行実行
   - ネストプランをサポート：サブプランを再帰的に実行（最大深度3）
   - 動的拡張：実行後にスキルを追加する必要があるか評価（最大1回）
   - コンテキスト渡し：後続のスキルは自動的に前のスキルの出力を`params["context"]`として受け取る

4. **エバリュエーター** (`evaluator.py`)
   - パイプラインが自然に終了した後、出力が十分かどうかを判断
   - 軽量モデル（llama-3.1-8b-instant）を使用
   - 不足している情報を補うために1つのスキルを追加することを推奨可能
   - 無限ループを防止（各プランは最大1回拡張）

5. **Groqクライアント** (`groq_client.py`)
   - 5層モデル自動フォールバックメカニズム：
     - Tier 1: openai/gpt-oss-120b
     - Tier 2: openai/gpt-oss-20b
     - Tier 3: qwen/qwen3-32b
     - Tier 4: meta-llama/llama-4-scout-17b-16e-instruct
     - Tier 5: llama-3.1-8b-instant
   - TPD（1日あたりの制限）エラー → 次の層のモデルに切り替え
   - RPM（1分あたりの制限）エラー → 5秒待機後に再試行
   - 状態の永続化：`st.session_state`を使用して使い果たされたモデルを追跡

6. **スキルモジュール** (`skills/*/skill.py`)
   - 各スキルには`run(params)`ジェネレーター関数がある
   - `context`パラメータをサポートして上流スキル出力を受け取る
   - UI表示用にストリーミング出力結果
   - 自己完結型のエラー処理
   - 完全な中国語ローカライゼーション

7. **Streamlitアプリケーション** (`app.py`)
   - チャットインターフェースと履歴
   - リアルタイムスキル呼び出しインジケーター（現在のモデルを表示）
   - ドキュメントアップロード管理（サイドバー）
   - 折りたたみ可能な推論プロセスと結果表示
   - 統一されたフォントとスタイル
   - ドキュメントストレージの永続化（`st.session_state.document_store`）

## ⚠️ 重要な注意事項

### ドキュメントQ&A (document_qa) キーポイント

1. **ドキュメントストレージの永続化**
   - ドキュメントは`st.session_state.document_store`に保存
   - Streamlitモジュールのホットリロードによるデータ損失を回避
   - モジュールレベルのグローバル変数でデータを保存しない

2. **ドキュメントの重複アップロードを防止**
   - `st.session_state.uploaded_files`を使用してアップロード済みファイルを追跡
   - ファイル名とサイズから一意の識別子を生成
   - アップロード前に既に存在するかチェック

3. **中国語検索の最適化**
   - 多層スコアリングメカニズムを使用：
     - サブストリングマッチ（+100点）
     - 単一文字マッチ（+1点/文字）
     - 単語マッチ（+10点/単語）
   - 中英混合クエリをサポート

4. **ドキュメント渡しメカニズム**
   - app.pyはparamsを通じてdocument_storeをスキルに渡す
   - スキルはOptional[DocumentStore]パラメータを受け取る
   - アップロードと検索が同じストレージオブジェクトを使用することを確認

### ストックアナリシス (stock_analysis) キーポイント

1. **時価総額フォーマット**
   - >= 1T: `$X.XXX T`として表示
   - >= 1B: `$X.XX B`として表示
   - < 1B: `$X.XX M`として表示

2. **ニュース翻訳**
   - Groq LLMを使用して英語ニュースを簡体字中国語に翻訳
   - 元のソースリンクを保持

3. **AI分析**
   - システムプロンプトに「重複コンテンツを避ける」指示を含む
   - 各観点は一度だけ述べる

### ウェブサーチ (web_search) キーポイント

1. **結果翻訳**
   - Tavilyは英語結果を返す
   - Groq LLMを使用して簡体字中国語に翻訳
   - 段落フォーマットを保持

### コードジェネレーション (code_generation) キーポイント

1. **完全な中国語化**
   - すべてのプロンプトと説明は簡体字中国語を使用
   - コードコメントは簡体字中国語を使用
   - 次のステップの説明は簡体字中国語を使用

## 📝 新しいスキルの追加

### 1. スキルディレクトリを作成

```bash
mkdir skills/my_new_skill
```

### 2. `skill.md`を作成

```markdown
# Skill: my_new_skill

## 説明
スキルの簡単な説明

## トリガー条件
- このスキルをトリガーするタイミング
- このスキルを示すキーワード

## 非トリガー条件
- このスキルを使用しないタイミング

## パラメータ
- param1: str  # 説明
- param2: int  # 説明
- context: str (オプション)  # パイプラインモードで自動注入される上流出力

## 戻り値
スキルが返すコンテンツ
```

### 3. `skill.py`を作成

```python
from typing import Generator

def run(params: dict) -> Generator[str, None, None]:
    """
    スキルエントリー関数
    
    Args:
        params: 必要なパラメータを含む辞書
                - 必須パラメータはルーターによって抽出される
                - context: パイプラインモードで自動注入される上流スキル出力
        
    Yields:
        結果文字列（Markdownフォーマットをサポート）
    """
    # パラメータを取得
    param1 = params.get("param1")
    if not param1:
        yield "❌ エラー：'param1'パラメータが不足しています\n"
        return
    
    # 上流コンテキストを取得（パイプライン内の場合）
    context = params.get("context", "")
    
    # 実装コード
    yield f"💡 処理中：{param1}\n\n"
    
    if context:
        yield f"📋 上流出力を受信（{len(context)}文字）\n\n"
    
    try:
        # タスクを実行
        result = your_processing_logic(param1, context)
        yield f"✅ 結果：{result}\n"
    except Exception as e:
        yield f"❌ エラー：{str(e)}\n"
```

### 4. アプリケーションを再起動

スキルは自動的に検出されロードされます！

### 5. パイプラインサポートの説明

スキルをパイプラインで使用する必要がある場合：
- **上流出力を受信**: `params.get("context", "")`で取得
- **下流に渡す**: `yield`出力は自動的に下流スキルの`context`になる
- **パラレル実行**: スキルは独立しており、実行順序に依存しない

## 🛠️ トラブルシューティング

### 問題: "GROQ_API_KEY not set"
**解決策**: `.env`ファイルを作成してAPIキーを入力したことを確認

### 問題: "Skill not found"
**解決策**: スキルディレクトリに`skill.md`と`skill.py`の両方が含まれているか確認

### 問題: "Import error"
**解決策**: すべての依存関係がインストールされていることを確認: `pip install -r requirements.txt`

### 問題: "Tavily API error"
**解決策**: TAVILY_API_KEYが正しく、利用可能なクォータがあることを確認

### 問題: "すべてのモデルが1日の制限に達しました"
**解決策**:
1. 翌日のリセットを待つ（UTC時間）
2. システムは自動的に5層モデルフォールバックを試行
3. Groqコンソールで使用クォータを確認

### 問題: ドキュメントアップロード後に検索できない
**解決策**:
1. session_stateストレージを使用しているか確認
2. document_storeがparamsを通じてスキルに渡されていることを確認
3. アプリケーションを再起動して古いモジュールキャッシュをクリア

### 問題: ドキュメントが重複して蓄積される
**解決策**:
1. uploaded_files追跡メカニズムを使用
2. 「すべてのドキュメントをクリア」ボタンをクリック
3. ドキュメントを再アップロード

### 問題: 中国語検索マッチが失敗する
**解決策**:
1. ドキュメント内の具体的なキーワードを使用
2. 異なる表現を試す
3. ドキュメントコンテンツに関連情報が含まれているか確認

### 問題: パイプライン実行がスタックする
**解決策**:
1. 最大ネスト深度に達しているか確認（デフォルト3層）
2. スキルが例外をスローしているか確認
3. パラレル実行時にスレッドプールが正常か確認

## 🔀 パイプライン実行モード詳細

### シリアル実行（Serial）
ユーザーが「まずAをしてからBをする」という意図を明確に表現した場合に使用。

**例**：
- 「NVIDIAの最新ニュースを検索してからNVDA株を分析して」
- 「まずドキュメントを確認してからコードを生成して」

**実行フロー**：
```
ステップ1: web_search → 出力A
ステップ2: stock_analysis (出力Aをcontextとして受信) → 出力B
```

### パラレル実行（Parallel）
ユーザーが「同時に」、「並行して」、「一緒に」複数のタスクを実行することを明確に表現した場合に使用。

**例**：
- 「NVIDIAニュースを検索してNVDA株を同時に分析して」
- 「AAPLとTSLAの株価を並行して照会して」

**実行フロー**：
```
パラレルグループ: [web_search, stock_analysis] → 同時実行 → 結果を集約
```

### ネストプラン（Nested）
プラン内にサブプランをネストすることをサポート、最大深度は3層。

**例**：
```json
{
  "plan": [
    {"skill": "web_search"},
    {
      "plan": [
        {"skill": "stock_analysis"},
        {"skill": "code_generation"}
      ]
    }
  ]
}
```

### 動的拡張（Dynamic Extension）
パイプライン実行完了後、エバリュエーターが出力が十分かどうかを判断：
- 重要な情報が不足している場合 → 自動的に1つのスキルを追加
- 各プランは最大1回拡張、無限ループを防止
- ✨マークを使用して動的に追加されたスキルを表示

**例シナリオ**：
ユーザー：「AAPL株を分析して、情報が不足していたら検索して」
1. ルーターは`stock_analysis`のみを返す（単一スキル）
2. `stock_analysis`を実行完了
3. エバリュエーターがより多くのニュース情報が必要と判断
4. 自動的に`web_search`スキルを追加

## 🎚️ Groqモデルフォールバックメカニズム

### 5層モデルアーキテクチャ

システムはインテリジェントな5層モデルフォールバックを実装し、高可用性を確保：

| 層級 | モデル | 用途 | フォールバックトリガー |
|------|------|------|----------|
| Tier 1 | openai/gpt-oss-120b | 最高品質 | TPD使い果たし |
| Tier 2 | openai/gpt-oss-20b | 高品質 | TPD使い果たし |
| Tier 3 | qwen/qwen3-32b | 中品質 | TPD使い果たし |
| Tier 4 | meta-llama/llama-4-scout-17b-16e-instruct | 基本品質 | TPD使い果たし |
| Tier 5 | llama-3.1-8b-instant | 高速レスポンス | TPD使い果たし |

### エラー処理戦略

1. **TPD（Tokens Per Day）エラー**
   - 現在のモデルを使い果たしたとマーク
   - 即座に次の層のモデルに切り替え
   - UIにフォールバック警告を表示

2. **RPM（Requests Per Minute）エラー**
   - 5秒待機
   - 同じモデルで再試行
   - 層級を切り替えない

3. **すべてのモデル使い果たし**
   - フレンドリーな中国語エラーメッセージをスロー
   - ユーザーに明日再試行するよう促す

### 状態の永続化

- `st.session_state.exhausted_models`を使用して使い果たされたモデルを追跡
- 非Streamlit環境（バックグラウンドスレッドなど）ではモジュールレベル変数をフォールバックとして使用
- セッション期間中フォールバック状態を維持し、使い果たされたモデルの再試行を回避

### UIフィードバック

- スキル呼び出しバブルに現在使用中のモデルを表示
- モデルフォールバック時にトースト警告を表示：`⚠️ モデルフォールバック: Tier1 → Tier2`
- 推論プロセスで使用されたモデル名を表示

## 🧩 コンテキスト渡しメカニズム

### 自動注入
パイプラインエグゼキューターは自動的に前のステップの完全な出力を次のステップの`params["context"]`に注入：

```python
# ステップ1出力
"NVIDIAが新しいGPUを発表..."

# ステップ2が自動的に受信
params = {
    "ticker": "NVDA",
    "context": "NVIDIAが新しいGPUを発表..."  # 自動注入
}
```

### スキル実装
スキルはcontextを選択的に使用可能：

```python
def run(params: dict) -> Generator[str, None, None]:
    context = params.get("context", "")
    if context:
        # 上流情報を組み合わせて分析
        yield f"以下の背景情報に基づいて：\n{context}\n\n"
    # 実行を続ける...
```
