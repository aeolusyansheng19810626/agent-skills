# 🤖 ルーターエージェント - 統合チャットインターフェース

LLMがユーザーの意図を自動で判断し、最適なスキルモジュールへルーティングするインテリジェントなAIエージェントシステムです。統一されたチャットインターフェースで、スキル実行状況をリアルタイムで確認できます。

## 🌟 主な機能

- **🎯 インテリジェントルーティング**: ユーザー入力から最適なスキルを自動選択
- **🔧 リアルタイム表示**: 実行中のスキル名を「🔧 実行中: xxx_skill」として表示
- **📦 モジュラー設計**: skill.mdとskill.pyを作成するだけで新規スキルを簡単に追加
- **💬 Streamlit UI**: 美しくレスポンシブなチャットインターフェース
- **🔄 ストリーミング出力**: スキル実行結果をリアルタイムで表示
- **🌐 多言語対応**: 日本語、中国語、英語に対応
- **🎨 パープルグラデーションデザイン**: モダンなパープルグラデーションテーマと最適化されたUIレイアウト

## 🚀 クイックスタート

### 1. 必要な環境

- Python 3.8以上
- Groq APIキー（[取得はこちら](https://console.groq.com/keys)）
- Tavily APIキー（[取得はこちら](https://tavily.com/)）

### 2. インストール

```bash
# リポジトリのクローン
git clone <your-repo-url>
cd agent-skills

# 仮想環境の作成
python -m venv venv

# 仮想環境の有効化
# Windows:
venv\Scripts\activate
# macOS/Linux:
source venv/bin/activate

# 依存パッケージのインストール
pip install -r requirements.txt
```

### 3. 環境変数の設定

```bash
# 環境変数テンプレートのコピー
cp .env.example .env

# .envファイルを編集してAPIキーを設定
# GROQ_API_KEY=your_groq_api_key_here
# TAVILY_API_KEY=your_tavily_api_key_here
```

### 4. アプリケーションの起動

```bash
streamlit run app.py
```

ブラウザで `http://localhost:8501` が自動的に開きます

## 🔧 動作原理

### ルーターエージェントの処理フロー

```
ユーザー入力 → ルーターエージェント → Groq LLM → 意図認識 → 単一スキル/パイプラインを返す → パイプライン実行 → ストリーミング出力 → UI表示
                                                                     ↓
                                                             動的評価（必要に応じてスキルを追加）
```

### 主要コンポーネント

1. **スキルローダー** (`skill_loader.py`)
   - `skills/`ディレクトリを自動スキャン
   - `skill.md`ファイルを解析してメタデータを取得
   - スキルレジストリを構築
   - 動的なスキルインポート機能

2. **ルーターエージェント** (`router.py`)
   - 全スキルの説明をLLMプロンプトに注入
   - Groq LLMで意図を認識（5層モデルフォールバック対応）
   - 3つの形式で結果を返す：
     - 単一スキル：`{"skill": "name", "params": {...}}`
     - 直列パイプライン：`{"plan": [{"skill": "name1"}, {"skill": "name2"}]}`
     - 並列パイプライン：`{"plan": [{"parallel": [{"skill": "name1"}, {"skill": "name2"}]}]}`
   - 推論プロセスを完全に出力
   - マッチしないクエリのフォールバック処理

3. **パイプライン実行エンジン** (`pipeline.py`)
   - 直列実行：スキルを順番に実行し、コンテキストを自動で引き継ぐ
   - 並列実行：ThreadPoolExecutorで並行実行
   - ネスト対応：サブプランを再帰的に実行（最大深度3）
   - 動的拡張：実行後にスキル追加の必要性を評価（最大1回）
   - コンテキスト引き継ぎ：後続スキルは前のスキルの出力を`params["context"]`で受け取る

4. **評価エンジン** (`evaluator.py`)
   - パイプライン終了後、出力の充足性を判断
   - 軽量モデル（llama-3.1-8b-instant）を使用
   - 不足情報を補うため1つのスキル追加を推奨
   - 無限ループ防止（各プランは最大1回のみ拡張）

5. **Groqクライアント** (`groq_client.py`)
   - 5層モデル自動フォールバック機構：
     - Tier 1: openai/gpt-oss-120b
     - Tier 2: openai/gpt-oss-20b
     - Tier 3: qwen/qwen3-32b
     - Tier 4: meta-llama/llama-4-scout-17b-16e-instruct
     - Tier 5: llama-3.1-8b-instant
   - TPD（1日あたりの上限）エラー → 次の層のモデルへ切り替え
   - RPM（1分あたりの上限）エラー → 5秒待機後に再試行
   - 状態の永続化：`st.session_state`で使用済みモデルを追跡

6. **スキルモジュール** (`skills/*/skill.py`)
   - 各スキルは`run(params)`ジェネレーター関数を実装
   - `context`パラメータで上流スキルの出力を受け取る
   - UI表示用にストリーミング出力
   - 自己完結型のエラー処理
   - 多言語対応

7. **Streamlitアプリケーション** (`app.py`)
   - チャットインターフェースと履歴管理
   - リアルタイムスキル実行表示（現在のモデルも表示）
   - ドキュメントアップロード管理（サイドバー）
   - 折りたたみ可能な推論プロセスと結果表示
   - 統一されたフォントとスタイル
   - ドキュメントストレージの永続化（`st.session_state.document_store`）

## ⚠️ 重要な注意事項

### ドキュメントQ&A (document_qa) のポイント

1. **ドキュメントストレージの永続化**
   - ドキュメントは`st.session_state.document_store`に保存
   - Streamlitモジュールのホットリロードによるデータ損失を回避
   - モジュールレベルのグローバル変数は使用しない

2. **ドキュメントの重複アップロード防止**
   - `st.session_state.uploaded_files`でアップロード済みファイルを追跡
   - ファイル名とサイズから一意の識別子を生成
   - アップロード前に既存チェックを実施

3. **検索の最適化**
   - 多層スコアリング機構：
     - 部分文字列マッチ（+100点）
     - 単一文字マッチ（+1点/文字）
     - 単語マッチ（+10点/単語）
   - 多言語混合クエリに対応

4. **ドキュメント受け渡し機構**
   - app.pyはparamsを通じてdocument_storeをスキルに渡す
   - スキルはOptional[DocumentStore]パラメータを受け取る
   - アップロードと検索で同じストレージオブジェクトを使用

### 株式分析 (stock_analysis) のポイント

1. **時価総額のフォーマット**
   - >= 1T: `$X.XXX T`形式で表示
   - >= 1B: `$X.XX B`形式で表示
   - < 1B: `$X.XX M`形式で表示

2. **ニュースの翻訳**
   - Groq LLMで英語ニュースを翻訳
   - 元のソースリンクは保持

3. **AI分析**
   - システムプロンプトに「重複コンテンツを避ける」指示を含む
   - 各観点は一度のみ言及

### Web検索 (web_search) のポイント

1. **結果の翻訳**
   - Tavilyは英語結果を返す
   - Groq LLMで翻訳
   - 段落フォーマットを保持

### コード生成 (code_generation) のポイント

1. **多言語対応**
   - プロンプトと説明は選択言語で表示
   - コードコメントも選択言語で生成
   - 次のステップの説明も選択言語で表示

## 📝 新しいスキルの追加方法

### 1. スキルディレクトリの作成

```bash
mkdir skills/my_new_skill
```

### 2. `skill.md`の作成

```markdown
# Skill: my_new_skill

## 説明
スキルの簡単な説明

## トリガー条件
- このスキルを起動するタイミング
- このスキルを示すキーワード

## 非トリガー条件
- このスキルを使用しないタイミング

## パラメータ
- param1: str  # 説明
- param2: int  # 説明
- context: str (オプション)  # パイプラインモードで自動注入される上流出力

## 戻り値
スキルが返すコンテンツ
```

### 3. `skill.py`の作成

```python
from typing import Generator

def run(params: dict) -> Generator[str, None, None]:
    """
    スキルのエントリーポイント
    
    Args:
        params: 必要なパラメータを含む辞書
                - 必須パラメータはルーターが抽出
                - context: パイプラインモードで自動注入される上流スキルの出力
        
    Yields:
        結果文字列（Markdown形式対応）
    """
    # パラメータの取得
    param1 = params.get("param1")
    if not param1:
        yield "❌ エラー：'param1'パラメータが必要です\n"
        return
    
    # 上流コンテキストの取得（パイプライン内の場合）
    context = params.get("context", "")
    
    # 実装
    yield f"💡 処理中：{param1}\n\n"
    
    if context:
        yield f"📋 上流出力を受信（{len(context)}文字）\n\n"
    
    try:
        # タスクの実行
        result = your_processing_logic(param1, context)
        yield f"✅ 結果：{result}\n"
    except Exception as e:
        yield f"❌ エラー：{str(e)}\n"
```

### 4. アプリケーションの再起動

スキルは自動的に検出され、ロードされます！

### 5. パイプライン対応について

スキルをパイプラインで使用する場合：
- **上流出力の受信**: `params.get("context", "")`で取得
- **下流への引き継ぎ**: `yield`した出力は自動的に下流スキルの`context`になる
- **並列実行**: スキルは独立しており、実行順序に依存しない

## 🛠️ トラブルシューティング

### 問題: "GROQ_API_KEY not set"
**解決方法**: `.env`ファイルを作成し、APIキーが正しく設定されているか確認

### 問題: "Skill not found"
**解決方法**: スキルディレクトリに`skill.md`と`skill.py`の両方が存在するか確認

### 問題: "Import error"
**解決方法**: すべての依存パッケージがインストールされているか確認: `pip install -r requirements.txt`

### 問題: "Tavily API error"
**解決方法**: TAVILY_API_KEYが正しく、利用可能なクォータが残っているか確認

### 問題: "すべてのモデルが1日の上限に達しました"
**解決方法**:
1. 翌日のリセットを待つ（UTC時間基準）
2. システムは自動的に5層モデルフォールバックを試行
3. Groqコンソールで使用状況を確認

### 問題: ドキュメントアップロード後に検索できない
**解決方法**:
1. session_stateストレージが使用されているか確認
2. document_storeがparamsを通じてスキルに渡されているか確認
3. アプリケーションを再起動して古いモジュールキャッシュをクリア

### 問題: ドキュメントが重複して蓄積される
**解決方法**:
1. uploaded_files追跡機構を使用
2. 「すべてのドキュメントをクリア」ボタンをクリック
3. ドキュメントを再アップロード

### 問題: 検索でマッチしない
**解決方法**:
1. ドキュメント内の具体的なキーワードを使用
2. 異なる表現を試す
3. ドキュメントに関連情報が含まれているか確認

### 問題: パイプライン実行が停止する
**解決方法**:
1. 最大ネスト深度に達していないか確認（デフォルト3層）
2. スキルが例外をスローしていないか確認
3. 並列実行時のスレッドプールが正常か確認

## 🔀 パイプライン実行モード詳細

### 直列実行（Serial）
ユーザーが「まずAをしてからBをする」という意図を明確に表現した場合に使用。

**使用例**：
- 「NVIDIAの最新ニュースを検索してからNVDA株を分析して」
- 「まずドキュメントを確認してからコードを生成して」

**実行フロー**：
```
ステップ1: web_search → 出力A
ステップ2: stock_analysis (出力Aをcontextとして受信) → 出力B
```

### 並列実行（Parallel）
ユーザーが「同時に」、「並行して」、「一緒に」複数のタスクを実行することを明確に表現した場合に使用。

**使用例**：
- 「NVIDIAニュースを検索してNVDA株を同時に分析して」
- 「AAPLとTSLAの株価を並行して照会して」

**実行フロー**：
```
並列グループ: [web_search, stock_analysis] → 同時実行 → 結果を集約
```

### ネストプラン（Nested）
プラン内にサブプランをネスト可能、最大深度は3層。

**例**：
```json
{
  "plan": [
    {"skill": "web_search"},
    {
      "plan": [
        {"skill": "stock_analysis"},
        {"skill": "code_generation"}
      ]
    }
  ]
}
```

### 動的拡張（Dynamic Extension）
パイプライン実行完了後、評価エンジンが出力の充足性を判断：
- 重要な情報が不足している場合 → 自動的に1つのスキルを追加
- 各プランは最大1回のみ拡張、無限ループを防止
- ✨マークで動的に追加されたスキルを表示

**シナリオ例**：
ユーザー：「AAPL株を分析して、情報が不足していたら検索して」
1. ルーターは`stock_analysis`のみを返す（単一スキル）
2. `stock_analysis`を実行完了
3. 評価エンジンがより多くのニュース情報が必要と判断
4. 自動的に`web_search`スキルを追加

## 🎚️ Groqモデルフォールバック機構

### 5層モデルアーキテクチャ

システムはインテリジェントな5層モデルフォールバックを実装し、高可用性を確保：

| 層 | モデル | 用途 | フォールバックトリガー |
|------|------|------|----------|
| Tier 1 | openai/gpt-oss-120b | 最高品質 | TPD上限到達 |
| Tier 2 | openai/gpt-oss-20b | 高品質 | TPD上限到達 |
| Tier 3 | qwen/qwen3-32b | 中品質 | TPD上限到達 |
| Tier 4 | meta-llama/llama-4-scout-17b-16e-instruct | 基本品質 | TPD上限到達 |
| Tier 5 | llama-3.1-8b-instant | 高速レスポンス | TPD上限到達 |

### エラー処理戦略

1. **TPD（Tokens Per Day）エラー**
   - 現在のモデルを使用済みとしてマーク
   - 即座に次の層のモデルへ切り替え
   - UIにフォールバック警告を表示

2. **RPM（Requests Per Minute）エラー**
   - 5秒待機
   - 同じモデルで再試行
   - 層は切り替えない

3. **全モデル使用済み**
   - わかりやすいエラーメッセージを表示
   - ユーザーに翌日の再試行を促す

### 状態の永続化

- `st.session_state.exhausted_models`で使用済みモデルを追跡
- 非Streamlit環境（バックグラウンドスレッドなど）ではモジュールレベル変数をフォールバックとして使用
- セッション中はフォールバック状態を維持し、使用済みモデルの再試行を回避

### UIフィードバック

- スキル実行バブルに現在使用中のモデルを表示
- モデルフォールバック時にトースト通知を表示：`⚠️ モデルフォールバック: Tier1 → Tier2`
- 推論プロセスで使用されたモデル名を表示

## 🧩 コンテキスト渡しメカニズム

### 自動注入
パイプラインエグゼキューターは自動的に前のステップの完全な出力を次のステップの`params["context"]`に注入:

```python
# ステップ1出力
"NVIDIAが新しいGPUを発表..."

# ステップ2が自動的に受信
params = {
    "ticker": "NVDA",
    "context": "NVIDIAが新しいGPUを発表..."  # 自動注入
}
```

### スキル実装
スキルはcontextを選択的に使用可能：

```python
def run(params: dict) -> Generator[str, None, None]:
    context = params.get("context", "")
    if context:
        # 上流情報を組み合わせて分析
        yield f"以下の背景情報に基づいて：\n{context}\n\n"
    # 実行を続ける...
```
